# 任务拆解策略

OpenClaw 作为技术 PM，需要将用户需求拆解为 Claude Code 可执行的子任务。

## 拆解原则

### 1. 原子性

每个子任务应该是 Claude Code 可独立完成的最小有意义单元：
- ✅ "实现用户注册 API，包含参数校验和数据库存储"
- ❌ "实现整个用户系统"（太大）
- ❌ "添加一个 import 语句"（太小，不值得调度）

### 2. 依赖关系明确

子任务之间的依赖必须显式标注：

```
Task 1: 创建数据库表结构（无依赖）
Task 2: 实现 Repository 层（依赖 Task 1）
Task 3: 实现 Service 层（依赖 Task 2）
Task 4: 实现 Controller 层（依赖 Task 3）
Task 5: 编写单元测试（依赖 Task 2, 3）  ← 可与 Task 4 并行
```

### 3. 上下文充分

每个子任务的 prompt 必须包含足够的上下文：

```
你正在一个 Spring Boot 项目中工作。
项目使用 Java 8 + Maven + MyBatis。
数据库是 MySQL 8.0。

当前任务: 实现用户注册 API

要求:
1. Controller 路径: POST /api/v1/users/register
2. 入参: username, password, email (都是必填)
3. 密码使用 BCrypt 加密存储
4. 邮箱唯一性校验
5. 返回 userId 和 cre建时间

现有代码结构:
- src/main/java/com/example/controller/
- src/main/java/com/example/service/
- src/main/java/com/example/mapper/
- src/main/java/com/example/entity/

请遵循阿里巴巴 Java 开发规范。
```

### 4. 完成标准清晰

每个子任务必须有明确的验收条件：

```
完成标准:
1. API 可正常调用并返回正确响应
2. 参数校验覆盖所有必填字段
3. 重复邮箱返回 409 Conflict
4. 单元测试覆盖正常和异常场景
5. 代码通过 mvn compile 编译
```

## 拆解模板

### 新功能开发

```
1. [数据层] 创建/修改数据模型和数据库表
2. [持久层] 实现 Repository/Mapper
3. [业务层] 实现 Service 层业务逻辑
4. [接口层] 实现 Controller/API 接口
5. [测试] 编写单元测试和集成测试
6. [验证] 整体编译和测试运行
```

### Bug 修复

```
1. [分析] 复现 bug，定位根因
2. [修复] 实现最小修复
3. [测试] 添加回归测试
4. [验证] 确认修复不引入新问题
```

### 重构

```
1. [分析] 识别代码坏味道，制定重构计划
2. [测试] 确保现有测试充分覆盖（不充分则先补充）
3. [重构] 小步执行重构（每步保持可编译可运行）
4. [验证] 运行全部测试确认行为不变
```

### 前端功能

```
1. [组件] 创建/修改 UI 组件
2. [状态] 实现状态管理逻辑
3. [API] 对接后端 API
4. [样式] 实现响应式布局和视觉效果
5. [测试] 组件测试和交互测试
```

## 并行策略

无依赖关系的子任务可以并行执行：

```
               ┌── Task A (后端 API) ──┐
需求分析 ──→ ├── Task B (前端组件) ──├──→ 集成测试 → 完成
               └── Task C (数据库迁移)──┘
```

并行上限建议: **3 个同时执行的 Claude Code 实例**（避免资源争用）

## 异常处理策略

| 异常类型 | OpenClaw 动作 |
|---------|--------------|
| 编译错误 | 读取错误日志，生成修复指令，重新派发给 Claude Code |
| 测试失败 | 分析失败原因，提供测试上下文，指导 Claude Code 修复 |
| 依赖冲突 | 提供正确的依赖版本信息，指导解决 |
| 执行超时 | 检查输出判断是否卡死，必要时 kill 并简化任务重试 |
| 文件冲突 | 基于需求理解决定保留哪个版本，指导合并 |
| 架构分歧 | 升级给用户决策（OpenClaw 提供分析和建议） |
